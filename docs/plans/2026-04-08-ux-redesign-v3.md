## 1. 设计核心理念

**一句话：冰箱里有什么、今晚吃什么、之前吃了什么。**

三个核心问题对应三个 Tab，第四个 Tab 是设置。不啰嗦、不堆砌、不长页面。

| 原则 | 说明 |
|------|------|
| 紧凑 | 每个页面一屏能看完核心内容，不要长滚动列表 |
| 主动 | App 主动告诉用户该做什么，不要让用户去找 |
| 温暖精致 | 圆角、渐变、微动画、emoji，有生活感不是冷冰冰的工具 |

## 2. 导航架构

### 2.1 TabBar

```
┌──────────┬──────────┬──────────┬──────────┐
│  🧊 冰箱  │  🍳 吃什么 │  📖 记录  │  ⚙ 我的  │
```

| Tab | 页面路径 | 一句话定位 |
|-----|---------|-----------|
| 冰箱 | `pages/fridge/index` | 看库存、知临期、快录入 |

| 记录 | `pages/meal-log/index` | 时间轴瀑布流，按日期展示吃过什么、照片 |
| 我的 | `pages/profile/index` | 设置冰箱层数/排序、管理调料、其他设置 |

### 2.2 完整页面地图

```

├── meal-log/

│
├── profile/

│   ├── condiments         ← 调料管理
│   └── shelves            ← 层架管理（层数 + 排序）
```

### 3.1 设计目标

一屏看完核心信息，不需要长滚动。录入要快，临期要显眼。

### 3.3 关键交互

**拍照入口：**
- 顶部三个卡片最右边是"拍照"卡片，点击进入拍照流程
- 拍照流程同 V2 文档描述（全屏拍照 → AI 识别 → 半屏确认 → 入库 → 回首页）

**常买食材快捷按钮：**
- 首页底部横向展示 4-6 个常用食材按钮（如鸡蛋、牛奶、上海青）
- 点击后弹出简易表单（只有数量和单位），一键入库
- 数据来源：用户最近添加频率最高的食材，初次使用显示默认列表

**层架点击：**
- 点击某层 → 跳转 `fridge/detail` 查看该层全部食材
- 每层卡片上如果有临期食材，显示橙色小圆点 + 数字

**临期区域：**
- 有临期食材时显示，没有时不显示（不要占空白区域）
- 点击临期食材名 → 跳转"吃什么"Tab 并自动推荐用掉该食材的菜谱

### 4.3 关键交互

**首次进入：**
- 自动触发一次 AI 推荐（不需要用户先输入）
- AI 根据当前库存 + 调料 + 临期情况，主动给出推荐
- 如果库存为空，提示"先去冰箱页录入食材"

**对话式推荐：**
- 用户可以追问："来个低脂的"、"不用番茄"、"10 分钟能搞定的"
- 每次追问 AI 会在之前上下文基础上调整推荐
- 对话历史保留在当前会话中

**菜谱卡片：**
- 嵌入在对话气泡中
- 紧凑显示：菜名 + 时间 + 用到什么食材 + 是否齐全
- 点击卡片展开详情（在当前对话中展开，不跳转）
- 详情包含：食材用量列表 + 烹饪步骤 + "开始烹饪"按钮

**"开始烹饪"：**
- 弹出确认："将扣减 番茄×2 鸡蛋×3"
- 确认后扣减库存 + 自动记录到饮食记录（Tab 3）
- Toast："烹饪愉快！已扣减食材 🎉"

**快捷提问：**
- 横向滚动的标签按钮
- "临期优先"：推荐能消耗临期食材的菜
- "10分钟快手"：时间 <= 10 分钟的菜
- "再来一批"：换一组推荐

**购物清单：**
- 当推荐菜谱缺少食材时，菜谱卡片上显示"缺: 面条 1把"
- 点击"加入购物"→ 加入购物清单
- 购物清单作为子页面从右上角图标进入（不是独立 Tab）

### 5.3 关键交互

**记录来源：**
1. **自动记录**：用户在"吃什么"Tab 点了"开始烹饪"→ 自动生成一条记录
2. **拍照记录**：用户做完饭后拍了成品照片 → 可以附加到记录上
3. **手动记录**：点击"手动记录"→ 弹出简易表单（吃了什么 + 照片 + 备注）

**每条记录包含：**
- 菜名（或自定义标题）
- 成品照片（0-3 张，横向排列，点击可放大查看）
- 用了哪些食材（如果有扣减记录）
- 餐类型标签（早/午/晚/夜宵/外食）
- 时间

**照片上传：**
- 在"开始烹饪"扣减完成后，弹出"拍一张成品照？"
- 或在记录详情中追加照片
- 照片存储在 Supabase Storage

**空状态：**
- 新用户："还没有记录，去推荐页做第一道菜吧"
- 配一张温暖的插画

**时间轴视觉：**
- 左侧有一条竖线连接所有记录（时间轴线）
- 日期分割用横线 + 文字居中
- 整体有"手账/日记"的温暖感，不是冷冰冰的列表

### 6.3 冰箱设置页 (`profile/shelves`)

**可做的事：**

## 7. 全局交互规范

### 7.1 空状态

| 页面 | 空状态 |
|------|--------|
| 冰箱页 | "冰箱空空的" + [拍照录入] 按钮 |
| 吃什么页 | "先去冰箱页录入食材" + [去录入] 按钮 |
| 记录页 | "还没记录，做第一道菜吧" + [去推荐] 按钮 |

### 7.2 加载态

- 统一骨架屏
- 拍照识别中：脉冲动画 + "正在识别..."
- 菜谱生成中：三个点跳动 + "小冰正在想..."

### 7.3 反馈

- 食材入库：Toast "已录入 N 样 🎉"
- 烹饪扣减：Toast "烹饪愉快！已扣减食材"
- 删除食材：Toast "已移除 XX"
- 操作失败：Toast "出了点问题，再试一次"

### 7.4 下拉刷新

冰箱页、吃什么页、记录页都支持。

## 8. 后端 API 需要配合的

| 变更 | 说明 | 优先级 |
|------|------|--------|
| 推荐接口支持对话上下文 | 传入历史消息，返回基于上下文的推荐 | P0 |
| 推荐接口默认考虑临期 | 返回结果优先推荐能消耗临期食材的菜 | P0 |
| 推荐接口支持换一批 | `seed` 或 `offset` 参数 | P0 |
| 饮食记录支持照片 | `POST /meals` 支持 `photo_urls: string[]` | P0 |
| 照片上传到记录 | `POST /meals/:id/photos` | P1 |
| 用户食材历史 | `GET /items/history` 常买食材列表 | P1 |
| 微信订阅消息 | 临期推送 | P2 |
| 统计接口 | `GET /stats/weekly` | P2 |
