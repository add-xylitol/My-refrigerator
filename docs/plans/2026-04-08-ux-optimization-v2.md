## 1. 用户画像（已确认）

**核心用户：25-35 岁城市上班族，独居或小家庭（1-2 人）**

| 特征 | 描述 |
|------|------|
| 冰箱规模 | 不大，10-20 样食材 |
| 做饭频率 | 每周 3-5 次，其余点外卖 |
| 买菜频率 | 每周 1-2 次 |
| 核心痛点 | 不知道冰箱里有什么、不知道该做什么、食材过期浪费 |
| 技术水平 | 会用小程序，不愿意学习复杂操作 |
| 使用频率 | 不是每天，每周 3-5 次关键场景 |

**用户不是来"管理冰箱"的，是来解决"今晚吃什么"的。**

## 2. 导航架构变更

### 2.1 当前（V1）

```
TabBar: 拍照 | 冰箱 | 发现 | 我的
```

问题：拍照是"动作"不是"页面"，不应该占据一个 Tab。

### 2.2 优化后（V2）

```
TabBar: 冰箱 | 发现 | 我的

拍照不是一个 Tab，而是一个全局浮动按钮（FAB）。
在冰箱页和发现页都能看到这个按钮。
```

**具体期望：**

- **Tab 1 冰箱**（首页）：用户打开小程序第一眼看到冰箱全貌和今日建议
- **Tab 2 发现**：菜谱推荐、购物清单、饮食记录
- **Tab 3 我的**：调料管理、层架管理、设置
- **拍照浮动按钮**：在冰箱页和发现页右下角，醒目的圆形大按钮（带相机图标），点击进入拍照流程。拍照完成后自动回到冰箱页

### 2.3 拍照流程的期望

拍照不再是一个"常驻页面"，而是一个"流程"：

```
用户点击拍照 FAB
  → 全屏拍照界面（相机取景框 + 拍照/相册两个按钮）
  → 拍完照后 AI 识别
  → 识别结果半屏弹出（从底部滑出）
  → 用户确认/调整
  → 入库 → 自动回到冰箱页
```

### 2.4 页面地图更新

```
pages/
├── fridge/
│   ├── index              ← Tab 1 首页（冰箱总览 + 今日建议 + 临期提醒）
│   ├── detail             ← 层架详情（该层食材列表）
│   ├── add                ← 手动添加/编辑食材
│   └── camera             ← 拍照流程页（从 FAB 进入，不再作为 Tab）
│
├── discover/
│   ├── index              ← Tab 2（菜谱推荐）
│   ├── recipe             ← 菜谱详情 + 开始烹饪
│   ├── shopping           ← 购物清单
│   └── meal-log           ← 饮食记录
│
├── profile/
│   ├── index              ← Tab 3（个人中心）
│   ├── condiments         ← 调料管理
│   └── shelves            ← 层架管理
```

**变更点：**
- `pages/camera/index` 移到 `pages/fridge/camera`
- TabBar 从 4 个减少为 3 个
- 冰箱页成为首页

### 3.3 新增模块："今日建议"

**期望行为：**
- 页面加载时自动生成 1-3 条建议，基于临期食材 + 用户库存
- 每条建议格式：「食材名 X天后到期，推荐做 XXX」
- 点击"做这道菜"→ 跳转菜谱详情页
- 点击"换一个"→ 滑动到下一条建议或重新请求 AI
- 没有临期食材时，显示通用建议：「冰箱里有 X 样食材，看看今天做什么」→ 点击跳发现页

**数据来源：**
- 后端根据 items 中 exp_date <= 3 天的食材 + AI 推荐匹配菜谱
- 或前端本地根据临期食材调用推荐接口

## 4. 拍照流程优化

### 4.1 当前问题

- 拍照是一个独立 Tab 页，占了一个导航位置
- 识别后"编辑修改"按钮没有实现功能
- 识别失败后只有一个小文字按钮"手动添加"
- 新用户不知道该拍冰箱全景还是拍单个食材

### 4.2 期望的拍照流程

**入口：**
- 冰箱页右下角 FAB 大按钮（相机图标，带脉冲动画提示新用户）
- 发现页顶部也可以有"拍照更新库存"入口
- 层架详情页空状态中的"去拍照"按钮

**流程：**

```
点击拍照 FAB
  → 进入全屏拍照页
  → 顶部有简短引导文字："拍一张冰箱内部的照片"
  → 中间是取景框
  → 底部两个大按钮：[📷 拍照] [🖼 从相册选择]
  → 拍照后：
     → 显示照片预览（可重拍）
     → AI 识别中：显示"正在识别..."动画
     → 识别成功：
        → 半屏底部弹出候选列表
        → 每个候选：名称、数量（可调）、建议层位（可改）、建议保质期
        → 每个候选有 ✓ 确认按钮
        → 底部两个按钮：[一键全部入库] [逐个编辑]
        → "逐个编辑"展开每个候选的完整编辑界面（数量、单位、层位、到期日）
     → 识别失败：
        → 显示友好的错误提示："没认出来，试试拍得再清楚一些？"
        → 两个按钮：[重新拍照] [手动添加]
        → "手动添加"跳转到 fridge/add 页面
     → 入库成功：
        → 自动返回冰箱页
        → Toast："已录入 N 样食材 🎉"
```

**关键改进：**
- 识别结果用半屏弹出而不是占满整个页面，用户能感知到"回到了冰箱页"
- "编辑修改"按钮要真正能用，展开内联编辑界面
- 识别失败的引导要友好，不要只显示技术错误信息

- 没有"换一批"功能
- 底部子导航（购物清单/饮食记录）容易被忽略

### 5.3 菜谱封面图方案

由于 AI 生成的菜谱没有真实照片，需要一种替代方案：

**推荐方案：使用食材 emoji 组合 + 渐变背景卡片**

- 每张菜谱卡片顶部有一个彩色渐变区域
- 渐变上放置 1-3 个相关食材的 emoji（大号）
- 例如：番茄炒蛋 → 🍅🥚，蒜蓉炒青菜 → 🧄🥬
- 颜色根据主要食材变化（番茄=红橙渐变，青菜=绿色渐变）
- 不需要额外图片资源，纯 CSS + emoji 实现

### 5.4 默认推荐策略

**当前**：默认显示"全部"，需要用户手动选筛选。
**期望**：

| 场景 | 默认推荐 |
|------|---------|
| 有临期食材（≤3天） | 默认选中"临期优先" |
| 没有临期食材 | 默认选中"全部" |
| 库存为空 | 显示空状态："先录入一些食材，我来推荐菜谱" + 跳转拍照/手动添加 |

### 5.5 "换一批"功能

**期望行为：**
- 在推荐列表底部显示"换一批推荐"按钮
- 点击后带一个新的随机参数重新调用推荐接口
- AI 端应支持 `seed` 或 `page` 参数来返回不同结果

## 6. 新增功能需求

### 6.1 P0：微信订阅消息（临期提醒）

**为什么是 P0：** 如果用户不打开小程序就看不到临期提醒，那提醒功能形同虚设。微信小程序的订阅消息是唯一能主动触达用户的通道。

**期望行为：**

| 触发条件 | 推送内容 | 推送时间 |
|---------|---------|---------|
| 有食材 1 天后到期 | "XX 明天就到期了，推荐做 XXX" | 每天 17:00（下班前） |
| 有食材今天到期 | "XX 今天到期！赶紧用掉吧" | 每天 9:00（早上） |

**实现方式：**
- 后端定时任务（Supabase pg_cron 或外部定时器）每天检查临期食材
- 调用微信订阅消息 API 推送
- 用户首次使用时引导授权订阅消息
- 用户可以在"我的"设置中开关提醒

**需要的微信消息模板：**
- 模板 1：临期提醒（食材名、到期日期、推荐做法）
- 模板 2：每日推荐（可选，每天推一条菜谱建议）

### 6.2 P1：常买食材记忆

**为什么：** 用户每周买的东西差不多。如果每次添加都要拍照或手动输入，门槛太高。

**期望行为：**
- 手动添加食材时，名称输入框下方显示"最近添加"列表（横向滚动）
- 例如："鸡蛋、牛奶、上海青、西红柿、豆腐"
- 点击直接填充名称 + 数量 + 单位 + 保质期（使用上次填的值）
- 数据来源：取用户最近 30 天添加过的食材，按频率排序
- 最多显示 8 个

### 6.3 P1：食材视觉强化

**为什么：** 纯文字的食材列表浏览效率低。用户扫一眼应该能识别出每种食材。

**期望行为：**
- 每种食材显示对应的 emoji 图标（已有部分实现，需要完善覆盖面）
- 临期食材卡片左边框用颜色条标识：红（已过期）、橙（1-2天）、黄（3-5天）、绿（5天以上）
- 食材 emoji 应覆盖至少 50 种常见食材

**常见食材 emoji 对照表（扩展）：**

```
蔬菜类: 🥬青菜 🥕胡萝卜 🍅番茄 🥒黄瓜 🌽玉米 🫑青椒 🧅洋葱 🥦西兰花 🍆茄子 🥔土豆 🧄大蒜 🍄蘑菇
水果类: 🍎苹果 🍌香蕉 🍊橙子 🍇葡萄 🍓草莓 🫐蓝莓 🍋柠檬
肉类: 🥩牛肉 🍗鸡肉 🐷猪肉 🐟鱼 🦐虾 🥓培根 🍖排骨
蛋奶: 🥚鸡蛋 🧀奶酪 🥛牛奶 🧈黄油
主食: 🍚米饭 🍜面条 🍞面包
饮料: 🍺啤酒 🥤饮料 🫖茶
调味: 🧂盐 🫒橄榄油 🍯蜂蜜
其他: 🧊冰块 🥜坚果 🫘豆腐
```

### 6.4 P1："没做饭"标记

**为什么：** 用户叫外卖或出去吃的时候，冰箱里的食材没有被消耗。如果没有这个标记，临期提醒的时间线会不准确。

**期望行为：**
- 在冰箱首页增加一个"今天没做饭"快捷按钮（可以放在今日建议卡片的替代位置）
- 点击后：记录到饮食记录（标记为"外食/外卖"），但不扣减任何库存
- 作用：系统知道用户今天没消耗食材，临期提醒更准确

### 6.5 P2：批量操作

**为什么：** 用户清理冰箱时，往往要一次扔掉/用完好几样东西。

**期望行为：**
- 层架详情页增加"管理"模式按钮
- 点击后进入多选模式：每个食材卡片左侧出现勾选框
- 底部出现批量操作栏：[删除选中] [移动到其他层架]
- 选择后执行操作

### 6.6 P2：用量统计（周报）

**为什么：** 用户想知道"我这个月浪费了多少"，形成反馈闭环，激励更好的库存管理。

**期望行为：**
- 在"我的"页增加"我的数据"入口
- 展示简单统计卡片：
  - 本周消耗：X 样
  - 本周浪费（过期删除）：X 样
  - 节省率：XX%（1 - 浪费/总消耗）
- 数据来源：recipe_usages 表 + 删除记录
- 可以做得非常简单，几个数字 + 环形图即可

## 7. 交互细节优化

### 7.1 空状态设计

**每个空状态都应该有：引导文案 + 行动按钮**

| 页面 | 空状态文案 | 行动按钮 |
|------|-----------|---------|
| 冰箱页（空冰箱） | "冰箱还是空的，拍一张开始吧" | [拍照录入] |
| 层架详情（空层架） | "这层还是空的" | [去拍照] [手动添加] |
| 发现页（无菜谱） | "先录入一些食材，我来推荐菜谱" | [去录入] |
| 购物清单（空） | "购物清单空空的，去发现页添加吧" | [去看菜谱] |
| 饮食记录（空） | "还没有记录，做完一道菜后会自动记录" | [去看菜谱] |

### 7.2 加载状态

**统一使用骨架屏，不要用菊花转圈。**

- 菜谱卡片加载：显示 2-3 个灰色矩形骨架
- 层架加载：显示 3-4 个带圆角的灰色条
- 拍照识别中：脉冲光圈 + 文字"正在识别..."

### 7.3 反馈动画

| 操作 | 期望动画 |
|------|---------|
| 食材入库成功 | 卡片从右侧滑入列表 + 绿色闪光 |
| 删除食材 | 卡片向左滑出 + 淡出 |
| 开始烹饪扣减 | 食材卡片依次飞出 + "烹饪愉快" 动画 |
| 标记购物已买 | 打勾动画 + 飞入"已买"区域 |

### 7.4 下拉刷新

**所有列表页都支持下拉刷新：**
- 冰箱页（刷新层架和库存数据）
- 发现页（刷新菜谱推荐）
- 购物清单
- 饮食记录
- 层架详情

## 8. 新用户引导

### 8.1 首次打开小程序

**期望流程：**

```
首次打开
  → 微信静默登录
  → 创建默认层架
  → 冰箱页显示新用户引导遮罩（3 步，可跳过）:

  Step 1: 指向右下角拍照 FAB
          "拍一张冰箱照片，AI 自动识别食材"
          [下一步]

  Step 2: 指向层架区域
          "点击层架，查看和管理每一层的食材"
```

## 9. 后端需要配合的变更

以下变更需要后端 API 支持才能实现上述 UX 优化：

| 变更 | 说明 | 优先级 |
|------|------|--------|
| 推荐接口增加 `seed` 参数 | 支持返回不同批次的推荐，用于"换一批" | P0 |
| 推荐接口默认按临期排序 | 后端根据 items 的 exp_date 自动排序 | P0 |
| 订阅消息后端任务 | 定时检查临期食材 + 调微信 API 推送 | P0 |
| 用户食材历史接口 `GET /items/history` | 返回用户常添加的食材名称和频率，用于"常买记忆" | P1 |
| 统计接口 `GET /stats/weekly` | 返回本周消耗/浪费数量 | P2 |
| 外食记录接口 | `POST /meals` 支持 `type: "eating_out"` | P1 |
