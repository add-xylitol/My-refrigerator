## 1. 设计原则

| 原则 | 说明 |
|------|------|
| 温暖精致 | 界面有生活气息，不是冷冰冰的数据库。圆角卡片、柔和渐变、微动画反馈 |
| 最多 3 步 | 从任何入口到达任何核心功能不超过 3 次点击 |
| 先做后选 | 拍照不需要前置选择层位，先拍再说 |
| 状态驱动 | 空冰箱、临期、缺货各有对应的引导和推荐，不做静态页面 |

## 2. 导航架构

### 2.1 底部 TabBar（4 个入口）

```
┌──────────┬──────────┬──────────┬──────────┐
│  📸 拍照  │  🧊 冰箱  │  🍳 发现  │  👤 我的  │
└──────────┴──────────┴──────────┴──────────┘
```

| Tab | 页面路径 | 职责 |
|-----|---------|------|
| 拍照 | `pages/camera/index` | 拍照 → AI 识别 → 半自动确认 → 入库 |
| 冰箱 | `pages/fridge/index` | 3D 层叠冰箱可视化 + 临期/缺货提醒 + 状态摘要 |
| 发现 | `pages/discover/index` | 菜谱推荐 + 购物清单 + 饮食记录 |
| 我的 | `pages/profile/index` | 个人信息 + 调料管理 + 层架管理 + 设置 |

### 2.2 完整页面地图

```
pages/
├── camera/
│   └── index              ← Tab 1: 拍照录入主页
│
├── fridge/
│   ├── index              ← Tab 2: 冰箱总览（3D层叠 + 提醒）
│   ├── detail             ← 点击某层架 → 该层食材列表
│   └── add                ← 手动添加食材表单
│
├── discover/
│   ├── index              ← Tab 3: 菜谱推荐列表
│   ├── recipe             ← 菜谱详情 + 开始烹饪按钮
│   ├── shopping           ← 购物清单
│   └── meal-log           ← 饮食记录历史
│
├── profile/
│   ├── index              ← Tab 4: 个人中心
│   ├── condiments         ← 调料管理（CRUD）
│   └── shelves            ← 层架管理（增删改排序）
│
└── index/                 ← 小程序启动页（可选启动动画后跳 camera）
```

### 2.3 页面跳转关系

```
拍照页 ────确认完成────→ 冰箱页（Toast: "已录入 N 样食材"）
冰箱页 ────点击层架────→ 层架详情页
冰箱页 ────点击临期食材─→ 发现页（带筛选：用掉该食材的菜谱）
冰箱页 ────点"+"按钮───→ 手动添加页
冰箱页 ────点"拍照"按钮→ 拍照页
发现页 ────点菜谱卡片──→ 菜谱详情页
菜谱详情 ──开始烹饪────→ 冰箱页（扣减完成 Toast + 推荐下次菜谱）
菜谱详情 ──缺料加购物──→ 购物清单页
购物清单 ──标记已买────→ 提示"加入库存？"
我的页 ────调料管理────→ 调料页
我的页 ────层架管理────→ 层架页
```

## 3. 各页面详细设计

### 3.1 拍照页 (`pages/camera/index`)

**定位**：最高频入口。用户打开小程序的第一动作。

**布局（从上到下）**：

```
┌─────────────────────────────────┐
│  状态栏                          │
├─────────────────────────────────┤
│  上次录入摘要（可折叠）            │
│  "上次录入 5 样 → 冷藏层2"        │
├─────────────────────────────────┤
│                                 │
│       [ 照片预览区域 ]            │
│       默认显示相机取景框           │
│       拍照后显示拍摄的照片          │
│                                 │
├─────────────────────────────────┤
│                                 │
│  ┌───────┐  ┌───────┐          │
│  │ 📷拍照 │  │ 🖼相册 │  ← 大按钮│
│  └───────┘  └───────┘          │
│                                 │
├─────────────────────────────────┤
│  AI 识别结果区（拍照后出现）       │
│                                 │
│  ┌─────────────────────────┐    │
│  │ 🥬 上海青  1把  AI预估3天│    │
│  │ [层位: 冷藏层1 ▾] [确认✓]│    │
│  ├─────────────────────────┤    │
│  │ 🥚 鸡蛋    6个  AI预估20天│   │
│  │ [层位: 冷藏层2 ▾] [确认✓]│    │
│  ├─────────────────────────┤    │
│  │ 🍅 西红柿  2个  AI预估5天 │   │
│  │ [层位: 冷藏层1 ▾] [确认✓]│    │
│  └─────────────────────────┘    │
│                                 │
│  [一键全部入库]  [编辑修改]       │
└─────────────────────────────────┘
```

**交互流程**：

1. 用户点击"拍照"或"从相册选择"
2. 调用 `wx.chooseMedia` 获取照片
3. 照片上传到后端 → 后端调 AI 识别 → 返回候选食材列表
4. 每个候选显示：名称、数量、AI 预估保质期（见第 5 节）、层位选择器
5. 用户可逐个调整数量/层位/到期日，也可点"一键全部入库"使用 AI 默认值
6. 入库后 Toast 提示 + 跳转冰箱页

**关键细节**：
- 层位默认由 AI 根据食材类型推断（蔬菜→果蔬盒，肉类→冷冻层，饮料→冷藏层）
- AI 预估保质期可编辑（点击可改为手动选择日期）
- 识别中显示骨架屏动画 + 进度提示
- 识别失败：显示"手动录入"按钮作为降级入口

## 4. 交互规范

### 4.1 全局交互

| 交互 | 行为 |
|------|------|
| 页面返回 | 保持上一页滚动位置和表单状态 |
| 下拉刷新 | 所有列表页支持下拉刷新 |
| 空状态 | 每个列表为空时显示引导文案 + 行动按钮 |
| 加载态 | 骨架屏，不用转圈 |
| 错误态 | 简短中文提示 + 重试按钮 |
| 网络断开 | 全局 Toast "网络不给力，稍后再试" |

### 4.2 Toast 规范

| 场景 | Toast 内容 |
|------|-----------|
| 食材入库成功 | "已录入 N 样食材" |
| 烹饪扣减成功 | "烹饪愉快！已扣减 N 样" |
| 删除食材 | "已移除 XX" |
| 加入购物清单 | "已加入购物清单" |
| AI 识别失败 | "识别失败，请手动录入" |
| 网络错误 | "网络不给力，稍后再试" |

### 4.3 动画规范

| 场景 | 动画 |
|------|------|
| 层架卡片点击 | `transform: scale(0.95)` + 回弹 |
| 食材入库 | 卡片从右侧滑入 |
| 删除食材 | 卡片向左滑出 |
| AI 识别中 | 脉冲光圈 + 文字"正在识别..." |
| 页面切换 | 右侧推入（子页面），左侧推出（返回） |

## 5. AI 智能保质期预估

**功能**：用户录入食材时（拍照或手动），AI 根据食材名称 + 层架类型自动推算保质期天数。

**实现方式**：
- 后端维护一份食材保质期参考表（初始 50+ 常见食材）
- 拍照识别时：AI 返回候选食材的同时返回建议保质期天数
- 手动添加时：前端调后端接口查保质期参考，自动填入到期日
- 用户可随时手动修改到期日

**示例保质期参考表（冷藏层）**：

| 食材 | 冷藏天数 | 冷冻天数 |
|------|---------|---------|
| 鸡蛋 | 30 | - |
| 牛奶 | 7 | - |
| 鸡胸肉 | 2 | 180 |
| 猪肉 | 2 | 180 |
| 西红柿 | 7 | - |
| 上海青 | 3 | - |
| 豆腐 | 5 | - |
| 草莓 | 3 | - |

## 6. 数据库变更（相对现有 0001_init.sql）

### 6.1 新增表

**profiles（替代 anon_profiles，支持微信登录）**：

```sql
create table public.profiles (
  id uuid primary key default gen_random_uuid(),
  wx_openid text unique,
  wx_unionid text,
  phone text,
  nickname text not null default '小冰',
  avatar_url text,
  daily_ai_quota int not null default 20,
  created_at timestamptz not null default timezone('utc', now()),
  updated_at timestamptz not null default timezone('utc', now())
);
```

**meal_logs（饮食记录）**：

```sql
create table public.meal_logs (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid not null references public.profiles(id) on delete cascade,
  recipe_id uuid references public.recipes(id) on delete set null,
  title text not null,
  items_used jsonb not null default '[]'::jsonb,
  photo_url text,
  note text,
  eaten_at timestamptz not null default timezone('utc', now()),
  created_at timestamptz not null default timezone('utc', now())
);
```

**shopping_items（购物清单）**：

```sql
create table public.shopping_items (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid not null references public.profiles(id) on delete cascade,
  name text not null,
  qty numeric(10,2) not null default 1,
  unit public.quantity_unit not null default '个',
  source text,
  purchased boolean not null default false,
  created_at timestamptz not null default timezone('utc', now())
);
```

**shelf_life_refs（食材保质期参考表）**：

```sql
create table public.shelf_life_refs (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  category text,
  chill_days int,
  freeze_days int,
  produce_days int,
  created_at timestamptz not null default timezone('utc', now())
);
create unique index shelf_life_refs_name_idx on public.shelf_life_refs(name);
```

### 6.2 修改现有表的外键

所有原来引用 `anon_profiles(id)` 的表改为引用 `profiles(id)`。

### 6.3 保留不变的表

`shelves`, `items`, `photos`, `condiments`, `recipes`, `recipe_usages`, `sync_logs` 结构不变，仅外键指向更新。

## 8. 核心数据流

### Flow 1: 拍照 → 库存

```
用户拍照
  → wx.chooseMedia 获取图片
  → POST /photos/upload 上传到 Supabase Storage
  → POST /vision/recognize (传 photo_id)
  → 后端调 GLM-5V-Turbo 识别
  → 返回候选食材 [{name, qty, unit, shelf_life_days, suggested_shelf}]
  → 前端展示候选列表，用户确认/调整
  → POST /items/batch 批量入库
  → Toast + 跳转冰箱页
```

### Flow 2: 库存 → 菜谱

```
用户打开发现页
  → POST /recipes/suggest (后端自动查 items + condiments)
  → 后端构建 prompt: "冰箱里有 X, Y, Z，调料有 A, B，推荐菜谱"
  → 调 GLM 返回菜谱列表
  → 前端展示，标注"食材齐全"或"缺 N 样"
  → 用户点击菜谱 → 详情页
  → 点击"开始烹饪" → POST /recipes/consume
  → 后端扣减 items + 写 recipe_usages + 写 meal_logs
  → 成功 → 跳转冰箱页
```

### Flow 3: 菜谱 → 购物清单

```
菜谱详情页显示"缺: 面条 1把"
  → 用户点击"加入购物清单"
  → POST /shopping {name: "面条", qty: 1, unit: "把", source: "牛肉面菜谱"}
  → 购物清单页显示
  → 用户标记已买 → PATCH /shopping/:id {purchased: true}
  → 提示"是否加入库存？" → 若是 → POST /items
```

### Flow 4: 临期提醒驱动

```
冰箱页加载时
  → GET /items → 筛选 exp_date <= now() + 2天
  → 显示临期卡片: "牛奶明天到期 → 推荐: 做奶茶?"
  → 点击推荐 → 跳转发现页 (带筛选参数)
  → 发现页自动请求"用掉牛奶的菜谱"
```

## 10. 技术实现约束

### 小程序环境

| 约束 | 影响 |
|------|------|
| 包体积 ≤ 2MB（主包） | 非核心页面使用分包加载 |
| 不支持直接调第三方 API | 所有 AI 调用走后端 |
| `wx.chooseMedia` 替代 `<input type="file">` | 拍照/相册交互用小程序原生 API |
| 无 localStorage | 用 Taro.setStorageSync / 后端持久化 |
| 冷启动 ≤ 3s | 首页数据预请求 + 缓存 |

### Taro 组件映射（vs 当前 React DOM）

| 当前 React DOM | Taro 小程序 |
|---------------|------------|
| `<div>` | `<View>` |
| `<span>`, `<p>` | `<Text>` |
| `<img>` | `<Image>` |
| `<input>` | `<Input>` |
| `<button>` | `<Button>` |
| `<form>` | `<Form>` |
| CSS/Tailwind | Taro CSS (rpx 单位) / SCSS |
